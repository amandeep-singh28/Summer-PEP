****Loop Graph****

Definition:
A Loop Graph is a special type of Conditional Graph in which the execution can return to a previous node and repeat the workflow until a specified condition is satisfied.

Why it is used?
- To retry failed operations.
- To improve or refine AI-generated responses.
- To repeat a task until the desired result is achieved.

Working:
Start
  ↓
Node A
  ↓
Node B
  ↓
Condition
 ├── True  → END
 └── False → Back to Node A (Loop)

Example:
An AI generates an answer → another AI evaluates it.
- If the answer is good → END.
- If the answer is incorrect → Generate the answer again.

Common Use Cases:
- Self-correcting AI Agents
- RAG applications (retrieve more documents if needed)
- Retrying failed API calls
- Code generation and debugging
- Multi-step planning agents

Key Point:
Unlike a normal graph where execution moves only forward, a Loop Graph can revisit previous nodes until the stopping condition becomes true.

******** Parallel Graph (LangGraph) ********

Definition:
A Parallel Graph is a graph where multiple nodes are executed simultaneously (in parallel) after a single node. This helps reduce execution time by performing independent tasks at the same time.

Why use it?
- Faster execution
- Better resource utilization
- Useful when tasks do not depend on each other

How it works:
START
   │
   ▼
 Main Node
   ├────────► Node A
   ├────────► Node B
   └────────► Node C
        │
        ▼
   Results are merged
        │
        ▼
       END

Real-world Example:
A travel assistant receives a user query:
- Node A → Searches flights
- Node B → Searches hotels
- Node C → Checks weather

All three tasks run simultaneously, and their results are combined before sending the final response.

Key Points:
- Multiple nodes execute at the same time.
- Independent tasks can run in parallel.
- Results are merged into a shared state.
- Improves speed and efficiency.
- Commonly used in AI Agents, RAG pipelines, and multi-tool workflows.

-> Elaborated Notes (ChatGpt)
Why do we need thread_id in LangGraph?
Problem Statement:-
    Suppose you are building a chatbot.

    A user chats with the bot today and comes back tomorrow. You want the
    chatbot to continue the previous conversation instead of starting from
    scratch.

Question: How does LangGraph know which previous conversation belongs to
the user?
    Answer: We pass a unique identifier called thread_id.
The thread_id does NOT store the conversation. It only identifies which
conversation history should be loaded.

------------------------------------------------------------------------
Example
Session 1 (Today)

User: Hi
AI: Hello!

User: My name is Amandeep.
AI: Nice to meet you, Amandeep.

The conversation is saved with:
thread_id = “user1”

Storage:
thread_id = “user1”

Conversation History: - Hi - Hello! - My name is Amandeep. - Nice to
meet you, Amandeep.

Notice that the conversation is stored, not the thread_id. The thread_id
is simply attached to that conversation.

------------------------------------------------------------------------
Session 2 (Tomorrow)
The same user comes back.

User: What is my name?
We again pass:
thread_id = “user1”

LangGraph thinks:
“I received thread_id = ‘user1’. Let me check the stored conversations.”

It finds:
thread_id = “user1”

Conversation History: - Hi - Hello! - My name is Amandeep. - Nice to
meet you, Amandeep.

It loads this history and replies:
AI: Your name is Amandeep.

------------------------------------------------------------------------

What if we DON’T pass the thread_id?

User: What is my name?
No thread_id is provided.

LangGraph cannot identify which previous conversation belongs to the
user, so it starts a brand-new conversation.

Conversation History: Empty
AI: Sorry, I don’t know your name.

------------------------------------------------------------------------

What if we pass the WRONG thread_id?
Suppose we pass:
thread_id = “user2”

Storage:
user1 -> Conversation of Amandeep user2 -> No previous conversation

LangGraph opens the conversation for user2, which is empty.
AI: Sorry, I don’t know your name.

------------------------------------------------------------------------
Important Difference

Many beginners think:
thread_id stores the memory.
This is incorrect.

Actually:
thread_id only identifies the memory.

Think of it like this:
-   thread_id = Roll Number of a student
-   Memory = Student’s Report Card

The roll number does not contain the marks. It simply helps the school
find the correct report card.

Similarly:
-   thread_id tells LangGraph which conversation to load.
-   Memory contains the actual conversation history and user
    information.

-> Techniques to implement STM
There are several techniques to implement STM:
    (i) Conversation Buffer Memory
        Where we pass all
    (ii) Conversation Window Buffer Memory
        It stores only the last K conversations and remove others.
        Instead of last-k messages we store only last k-tokens
    (iii) Conversation Summary Memory
        Stores previous message as summary
    (iv) Conversation Summary Buffer Memory
        For the last n message, we will store as it is, beyond that we will store the summary.

Summarize node
    Will only summarize if summarize is needed. If suppose user has only posted 1 message than we will not summarize. If message count > 5 than we will summarize only

****Long Term Memory****(Cross thread)
(This categorization is for our reference, this is not in LangGraph)
There are three different sub categories in which we store LTM(Only the important information is stored):-
    (i)Episodic memory
        Whatever is the history of the chat is stored in there.
    (ii)Semantic memory
        All the user related facts(entity)
    (iii)Procedured memory

Summary Table
Memory Type	            Stores	                                Example
Episodic Memory	        Important events/experiences	        "User got placed at TCS Prime."
Semantic Memory	        Facts about the user	                "User's name is Amandeep."
Procedural Memory	    User's preferred way of interaction	    "Explain step-by-step."
